## Issues

### #A — Add admin send-message service (EmailService + audit integration)   [logic, risk:heavy]

## Summary
Add the backend capability that lets an admin send a message (a subject and a body) to a chosen set of member recipients. The service sends each recipient's copy through the app's shared email path (automatic retry-on-failure and delivery-failure logging), and records one audit entry for the admin's send action, the same way every other admin action in the app is audited. This is the send-and-audit capability the admin compose screen (a separate issue) calls into; it does not itself build the compose screen.

## Acceptance criteria
- [ ] Given a subject, body, and a non-empty set of valid `recipientIds`, the service sends the message to each recipient via `EmailService.sendWithRetry()` and records one `AuditLog` entry for the admin action, capturing who sent it, the recipient set, the subject, and when.
- [ ] Given an empty `recipientIds` set, the service rejects the request with a validation error before any send is attempted — no call into `EmailService`, no `AuditLog` entry recorded.
- [ ] Given a recipient's send exhausts `sendWithRetry()`'s retries, the failure is recorded in the shared `EmailDeliveryLog`, matching the app's standard delivery-failure path (surfaced on the existing admin delivery-failure screen with its resend action — this issue does not build a new screen).
- [ ] Given a multi-recipient send where some recipients succeed and others exhaust retries, each recipient's outcome is tracked independently: the admin action is still audited once via `AuditLog`, and only the failed recipients are recorded in `EmailDeliveryLog`.

## Design (recorded, consistent)
- New send-admin-message capability accepts `{subject, body, recipientIds}` and, for each recipient, calls the shared `EmailService.sendWithRetry()` (`src/services/EmailService.ts`) — retry-on-failure is the conventional path for every outbound send, not a per-feature decision.
- Provider config reuses `src/config/email.ts` — no new provider setup.
- A send that exhausts retries is recorded in the shared `EmailDeliveryLog` (`src/models/EmailDeliveryLog.ts`); the existing admin delivery-failure screen and its resend action are the standard companion surface for this outcome — this issue does not rebuild that screen.
- One `AuditLog` (`src/models/AuditLog.ts`) entry records the admin action (who sent it, the recipient set, the subject, and when) — "the same way every other admin action is audited." Per-recipient delivery outcome (success vs. exhausted-retries) is tracked separately in `EmailDeliveryLog`, not by creating multiple audit entries.
- Empty `recipientIds` is rejected up front — no send attempted, no audit entry recorded — consistent with the brief's "selected set of members" (a send always targets a chosen, non-empty set).
- Out of scope: suppression/opt-out policy is an explicitly open product decision (conventions.md's note) — this service sends to every recipient it is given and implements no suppression logic.
- Convention followed: `.project/conventions.md` — Outbound messaging (provider config reuse via `src/config/email.ts`), Delivery + retry (`EmailService.sendWithRetry()`), Delivery-failure log (`EmailDeliveryLog`), Audit (`AuditLog`).

### #B — Build admin compose-and-send screen   [ui, risk:heavy]

## Summary
Add a new admin screen that lets an admin compose and send an email message to a set of members: pick recipients from the existing member directory, write a subject and body, and send. The screen invokes the send-admin-message contract introduced by #A (subject/body/recipientIds via `EmailService.sendWithRetry()`) and reuses the app's existing `AdminFormLayout` for consistency with other admin compose/edit screens.

## Acceptance criteria
- [ ] Happy path: admin selects one or more recipients from the existing member directory/picker, enters a non-empty subject and body, taps Send, confirms in a summary step (recipient count + subject) — on confirm the screen invokes #A's send-admin-message contract with `(subject, body, recipientIds)` and shows a success confirmation.
- [ ] Empty state: on first load, with no recipients selected and Subject/Body empty, the screen renders its empty/default compose state (`.project/conventions.md#States`) with Send disabled.
- [ ] Error/failure path: if the send contract call fails or rejects, the screen surfaces an inline error per `.project/conventions.md#States`, preserves the admin's entered recipients/subject/body (no data loss), and leaves Send available to retry — retry-on-transient-failure itself is #A's `sendWithRetry()` concern, not re-implemented here.
- [ ] Disabled/edge state: Send is disabled until at least one recipient is selected and both Subject and Body are non-empty, and is disabled again while a send is in flight (prevents double-submit).

## Design (recorded, consistent)
- Layout: reuse the existing `AdminFormLayout` (`src/components/AdminFormLayout.tsx`) for this compose screen, consistent with the app's other admin compose/edit screens.
- Recipient selection: reuse the existing member directory/picker as-is — the brief marks "redesigning the member directory" out of scope, so this screen integrates with the picker that already exists elsewhere in the admin UI rather than building a new one.
- Fields: Subject (single-line, required) and Body (multi-line, required).
- Send action: invokes the send-admin-message contract introduced by #A — `EmailService.sendWithRetry()` — passing `subject`, `body`, and the selected `recipientIds`; this screen does not re-implement retry-on-failure, that is #A's contract concern.
- States: empty and error states are required (`.project/conventions.md#States`) — an initial empty compose state before any input, and an inline error on a failed send attempt that preserves the admin's entered data.
- Confirm affordance: because sending is irreversible and affects a bulk set of real members (risk: heavy), Send opens a confirm step summarizing the recipient count and subject before the contract is invoked; canceling returns to the compose form unchanged.
- Disabled state: Send is disabled until at least one recipient is selected and both Subject and Body are non-empty, and again while a send is in flight.
- Accessibility: the Recipients control keeps its existing accessible label from the member picker (not redesigned); Subject field labeled "Subject"; Body field labeled "Message body"; Send button labeled "Send message"; the confirm step's primary action labeled "Send to {n} recipients", secondary "Cancel"; the error message is exposed to assistive tech (e.g. an accessible alert/live region) per `.project/conventions.md#States`.
- Out of scope (per the brief): redesigning the member directory/picker. The delivery-failure log + resend screen, the send-provider configuration, and the outbound audit trail are companion surfaces owned elsewhere (see #A / sibling candidates), not this screen.
- Convention followed: `.project/conventions.md#UI` (AdminFormLayout reuse) and `.project/conventions.md#States` (empty + error states required).

### #D — Add send preview/confirmation step to compose-and-send   [ui, risk:light]   [implied — review / trim / augment]

## Summary
Before an admin's message is actually sent, show a preview/confirmation step that displays the exact subject, body, and recipient count the admin entered on the compose screen, and requires an explicit confirm before the send is invoked. Sending is effectively irreversible once accepted (every outbound message is audited and, on delivery failure, logged) so the admin gets one last, accurate look at what will go out and to how many people before it happens.

## Acceptance criteria
- [ ] Happy path: the admin reviews the exact subject, body, and recipient count carried over from the compose screen (#B) and taps Send; the values shown match #A's send payload contract (subject/body/recipientIds) before the send call is invoked.
- [ ] Empty state: when the compose screen (#B) carries zero selected recipients, the preview shows "0 recipients selected" and disables Send, with a message directing the admin back to the compose screen to pick recipients.
- [ ] Error / failure path: if invoking #A's send call fails, the preview surfaces an inline error message and keeps the admin on the preview screen with the subject/body/recipient values retained, so Send can be retried without re-entering data.
- [ ] Disabled / edge state: while #A's send call is in flight, Send is disabled and shows a sending indicator, preventing a duplicate send from repeat taps.

## Design (recorded, consistent)
- Placement: this screen sits between #B's compose screen and #A's send call — the admin reviews here before the send is invoked, not after.
- Data shown: subject, body, and recipient count carried verbatim from #B's compose screen, matching #A's send payload contract (subject/body/recipientIds) — no re-entry, no re-derivation of these values on this screen.
- UI pattern: reuse the existing `AdminFormLayout` so the preview reads as a continuation of #B's compose flow rather than a separate pattern.
- States: loading (Send disabled + sending indicator while #A's call is in flight), error (inline message on send failure; admin stays on the preview to retry), empty/edge (zero recipients selected on #B: show "0 recipients selected", disable Send, point back to #B), disabled (Send disabled both during send-in-flight and on the zero-recipient edge).
- Affordances: a primary "Send" action that is the confirm step and invokes #A's send call; a secondary "Back / Edit" action that returns to #B's compose screen without sending — required because the send is not reversible once accepted (every outbound message is audited, and a failed-after-retries send is logged for follow-up, not silently discarded).
- Accessibility: each reviewed field carries a paired accessible label ("Subject: <value>", "Body preview: <value>", "Recipients: N selected"); the Send button's accessible name states the recipient count (e.g., "Confirm and send to N recipients") so the action's scope is announced, not only shown visually; the Back/Edit affordance carries an accessible label distinct from Send to prevent mis-activation.
- Convention followed: `.project/conventions.md` — UI bullet ("reuse the existing AdminFormLayout ... for admin compose / edit screens") and States bullet ("empty and error states are required on every user-facing surface").

## Dependencies
