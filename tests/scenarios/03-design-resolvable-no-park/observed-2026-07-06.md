### #A — Build Account Settings profile fields section (name, email, avatar)   [ui, risk:light]
## Summary
Add a `ProfileFieldsSection` component that renders the account holder's editable name, email, and avatar, so the Account Settings page (built separately, see Dependencies) can compose it into the full settings surface. Name and email use the shared `FormField` component with its standard validate-on-blur behavior; avatar editing reuses the shared `ImageUploader` component as-is. This issue scopes only the fields section itself — the page layout, the sticky Save bar, and the save/rollback flow belong to the page shell that composes this section, not to this issue.

## Acceptance criteria
- [ ] Happy path: `ProfileFieldsSection` renders the account holder's current name, email, and avatar (name/email via `FormField`, avatar via `ImageUploader`) and lets the user edit each; editing name or email and blurring the field with a valid value shows no error for that field.
- [ ] Empty state: when the account has no name, email, or avatar set yet, each control renders its standard empty/placeholder presentation (`FormField` empty input, `ImageUploader` default placeholder) rather than blank or broken UI, per `.project/conventions.md#States`.
- [ ] Error / failure path: an invalid entry (e.g., malformed email, blank required name) shows `FormField`'s inline error text under the field on blur, per `.project/conventions.md#Forms`.
- [ ] Disabled / edge state: while an avatar upload is in progress, `ImageUploader` shows its own existing busy/disabled affordance for the avatar control; the name and email fields remain independently editable and are unaffected by the avatar upload's in-flight state.

## Design (recorded, consistent)
- Name and email fields are built with the shared `FormField` component, validating inline on blur with error text under the field — Convention followed: `.project/conventions.md#Forms` (pattern: `src/components/FormField.tsx`).
- Avatar editing reuses the shared `ImageUploader` component unmodified, including its own busy/disabled state during an in-flight upload — Convention followed: `.project/conventions.md#Avatar upload` (pattern: `src/components/ImageUploader.tsx`).
- Empty and error states are required on every user-facing surface, applied here to all three controls — Convention followed: `.project/conventions.md#States`.
- Scope boundary: the two-column layout, sticky Save bar, and optimistic-save/rollback flow (`.project/conventions.md#Settings pages`, pattern: `src/pages/AccountSecurityPage.tsx`) belong to the page shell that composes this section, not to this issue — not duplicated here to avoid two issues owning the same save behavior.

## Dependencies
- None. No incoming dependency for #A. A downstream candidate depends on this issue (it composes the `ProfileFieldsSection` component #A introduces); that edge is recorded on the downstream issue, not here.

### #B — Add notification preference toggles to the Account Settings page   [ui, risk:light]
## Summary
Add a notification-preferences section to the Account Settings page so a user can turn their existing notification preferences on or off, following the app's established settings-page pattern (optimistic save, success toast, rollback-on-error). This exposes the user's existing notification-preference schema as toggles — it does not introduce new preference types or new notification categories.

## Acceptance criteria
- [ ] Happy path: the user toggles a notification preference; the toggle flips immediately (optimistic update) and a success toast confirms the change saved, mirroring the AccountSecurityPage save behavior.
- [ ] Empty state: if the user has no notification preferences to show (e.g. the preference list comes back empty), the section renders the required empty state instead of a blank or broken section.
- [ ] Error / failure path: if a toggle's save request fails, the toggle rolls back to its prior value and an error is surfaced to the user, per the rollback-on-error pattern.
- [ ] Disabled / edge state: while a given toggle's save request is in flight, that toggle is disabled so a second toggle can't race it, and it re-enables once the request resolves (success or rollback).

## Design (recorded, consistent)
- The section mirrors the existing `AccountSecurityPage` settings-page pattern: optimistic save on each toggle change, a success toast on save, and rollback-on-error on failure.
- Empty and error states are both required and both rendered explicitly, not left blank.
- The toggle/switch primitive itself is an implementation-level choice — no project convention names a specific toggle component — and is not a product decision; whichever standard interactive toggle control the project already uses elsewhere is reused here.
- The toggle set surfaces the user's existing notification-preference schema; this issue does not define or invent new preference types.
- No destructive-action confirm affordance is needed: toggling a preference on/off is reversible, and the rollback-on-error path already covers the failure case — this is not a destructive operation.
- Accessibility: each toggle carries an accessible name (label or aria-label) that identifies the specific preference it controls, so a screen-reader user can tell which preference a given toggle is for.
- Convention followed: `.project/conventions.md#Settings pages` (mirror `AccountSecurityPage`'s optimistic-save/success-toast/rollback-on-error pattern) and `.project/conventions.md#States` (empty and error states required on every user-facing surface).

## Dependencies

### #C — Build Account Settings page shell (layout, states, section composition)   [ui, risk:heavy]
## Summary
Build the Account Settings page shell: a new route with a two-column layout and a sticky Save bar mirroring the existing `AccountSecurityPage` pattern, composing the previously-built ProfileFieldsSection (#A) and NotificationPreferencesSection (#B) into that layout, plus the page-level empty and error states required on every user-facing surface. This issue delivers layout, composition, and states only — the Save action's own save orchestration (the optimistic update / success toast / rollback network wiring) is out of scope here and lands in the downstream "Wire Account Settings save behavior" issue.

## Acceptance criteria
- [ ] Happy path: navigating to the Account Settings route renders a two-column layout mirroring `AccountSecurityPage`, with the ProfileFieldsSection (#A) and NotificationPreferencesSection (#B) each composed into their own column, and a sticky Save bar pinned to the bottom of the viewport.
- [ ] Empty state: while the page's settings data has not yet finished loading, the shell renders a page-level loading/empty placeholder in place of the two-column form.
- [ ] Error state: if the page's initial settings-data load fails, the shell renders a page-level error message with a retry affordance in place of the form.
- [ ] Disabled/edge state: the Save bar's Save action is disabled while the page's settings data has not finished loading, and disabled whenever neither composed section reports unsaved changes.

## Design (recorded, consistent)
- Layout: two-column form layout with a sticky Save bar pinned to the bottom of the viewport, mirroring the existing `AccountSecurityPage` pattern (`src/pages/AccountSecurityPage.tsx`, referenced not modified).
- Composition: the shell renders ProfileFieldsSection (#A) and NotificationPreferencesSection (#B) as the page's two columns; the shell owns layout, page-level data-readiness, and Save-bar orchestration only — each section owns its own field rendering/validation.
- Empty/error scope: the empty and error states recorded here cover the page's own initial settings-data load (per `.project/conventions.md#States`, required on every user-facing surface) — not the Save action's own success/failure handling, which is wired by the downstream save-behavior issue.
- Save-bar readiness: the Save action is disabled until the page's settings data has loaded, and disabled again whenever neither composed section reports unsaved changes — an aggregate of the two sections' own dirty state, since the shell is the composition point for both.
- Accessibility: the Save action is a labelled button (accessible name "Save changes"); the page-level error state's retry affordance is a labelled button (accessible name "Retry"); the loading/empty placeholder uses an appropriate status/announcement role so assistive-tech users are not left on a blank page during the initial load.
- Convention followed: `.project/conventions.md#Settings pages` (two-column layout + sticky Save bar, mirroring `AccountSecurityPage`); `.project/conventions.md#States` (empty + error states required on every user-facing surface).

### #D — Wire Account Settings save behavior (optimistic save, toast, rollback-on-error)   [ui, risk:heavy]
## Summary
Wire the sticky Save bar's save behavior for the Account Settings page: submitting a change optimistically applies the new values immediately, confirms success with a toast, and rolls back to the last-saved values with an error shown if the save request fails — mirroring `AccountSecurityPage`'s save mechanics per `.project/conventions.md#Settings pages`. This turns the Save bar #C introduces into a working save action for the profile fields (name, email, avatar) and notification toggles.

## Acceptance criteria
- [ ] Clicking Save on the sticky Save bar applies the changed profile fields and/or notification toggles optimistically (the UI reflects the new values immediately) and shows a success toast confirming the save.
- [ ] When there are no unsaved changes, the Save action is inactive and no save request is made.
- [ ] If the save request fails, the displayed values roll back to their last-saved state and an error is surfaced to the user.
- [ ] While a save request is in flight, the Save action is disabled so a second submit cannot be made until the first completes.

## Design (recorded, consistent)
- Save trigger: the sticky Save bar introduced by #C is the only save trigger for this page; this issue wires its click handler to the behavior below.
- Optimistic save: on Save, apply the edited values (name, email, avatar, notification toggles) to the displayed state immediately, before the save request resolves.
- Success: on a successful save response, keep the optimistically-applied values and show a success toast.
- Rollback-on-error: on a failed save response, revert the displayed values to their last-saved state and surface an error to the user (e.g. an error toast) so they can retry.
- In-flight disable: the Save action is disabled for the duration of an outstanding save request, preventing a duplicate submit — part of mirroring `AccountSecurityPage`'s save mechanics.
- No-op when clean: the Save action stays inactive whenever there are no unsaved field changes (nothing to send) — consistent with the dirty-state gating #C's sticky Save bar already provides.
- Accessibility: the Save control carries an accessible name (e.g. "Save changes"); its disabled state (no changes / save in flight) is exposed to assistive tech via the disabled/aria-disabled attribute; the success and error toasts are announced to screen readers (e.g. an ARIA live region), not conveyed by color or icon alone.
- Existing pattern to mirror: `AccountSecurityPage`'s save mechanics, `src/pages/AccountSecurityPage.tsx`.
- Convention followed: `.project/conventions.md#Settings pages` — "mirror the existing `AccountSecurityPage` layout — a two-column form, a sticky Save bar at the bottom, optimistic save with a success toast and rollback-on-error. Pattern: `src/pages/AccountSecurityPage.tsx`."
