## Original brief
# Brief: Admin list pages

Build the admin list pages. Each page shows a table of its records with the standard
columns for that entity, plus row actions (view / edit / delete).

Pages to build (one per entity):

1. Users
2. Orders
3. Products
4. Invoices
5. Shipments
6. Refunds
7. Coupons
8. Reviews
9. Support Tickets
10. Audit Log

In scope: the 10 list pages above.
Out of scope: the detail / edit pages behind the row actions (a separate milestone).
## End original brief

## Milestone description (Wave order)
Build ten per-entity admin list pages (Users, Orders, Products, Invoices, Shipments,
Refunds, Coupons, Reviews, Support Tickets, Audit Log), each a server-paginated,
sortable/filterable table built on the shared `DataTable` component per the standing
data-table convention (`.project/design-system.md#Design system — data tables
(standing convention)`), plus one implied cross-cutting search enhancement to the same
shared component. The Audit Log page (#J) is parked — needs product input (whether
audit-trail rows may carry destructible edit/delete row actions) — and is not part of
this build.

### #A — Build Users admin list page   [ui, risk:light]
## Summary
Add the Users admin list page, one of ten entity list pages in the admin-list-pages initiative (brief: "Each page shows a table of its records with the standard columns for that entity, plus row actions (view / edit / delete)"). The page renders User records in the shared DataTable component per the standing data-tables convention: server-side pagination, per-column sort/filter (except the Actions column), a skeleton loading state, and the shared EmptyState plus CTA when there are no users. View and Edit row actions render as disabled/stub controls, since the detail/edit pages are explicitly out of scope in this brief (deferred to a separate milestone); Delete is fully implemented inline (confirm dialog plus API call, no separate page).

## Acceptance criteria
- [ ] Happy path: navigating to the Users list page renders a DataTable of User records, paginated server-side at 30 rows/page with a page-size control defaulting to 30, using the User entity's existing fields as columns; every column is sortable and filterable except the Actions column.
- [ ] Happy path (row actions): each row shows View, Edit, and Delete controls. View and Edit render present but disabled, with an accessible label indicating they are not yet available (e.g. `aria-label="View user (coming soon)"` / `aria-label="Edit user (coming soon)"`) and cannot be focused into navigating anywhere. Delete renders enabled.
- [ ] Destructive action: activating Delete opens a confirm dialog identifying the target user (e.g. by name/email) before calling the delete API; the dialog is reachable by keyboard and traps focus while open. Canceling leaves the row unchanged; confirming calls the API and removes the row (or triggers a re-fetch reflecting the deletion).
- [ ] Empty state: when there are zero Users, the page shows the shared EmptyState illustration plus a primary CTA relevant to the Users entity, instead of the table.
- [ ] Loading state: while the page's data is loading, the DataTable shows skeleton rows instead of a blank page or the empty state.
- [ ] Error / failure path: when the Users list fails to load, the page surfaces an inline error state in place of the table/skeleton, and does not silently render an empty table or stale data.

## Design (recorded, consistent)
- Reuse the shared DataTable component (`src/components/DataTable.tsx`); no hand-rolled table.
- Columns are the User entity's existing/standard fields, as already defined by the app's current User data model — this mirrors the entity's established shape rather than introducing a new field set.
- Every column is sortable and filterable except the Actions column, which carries no sort/filter.
- Pagination is server-side at 30 rows per page, with a page-size control defaulting to 30.
- Empty state: shared EmptyState illustration plus a primary CTA scoped to Users (exact CTA copy, e.g. a create/invite action, is an implementation detail consistent with the entity's create affordance elsewhere in the app — not a new interaction to design).
- Loading state: skeleton rows while the page loads.
- View and Edit row actions render disabled/stub this milestone: the detail/edit pages they'd open are explicitly out-of-scope per the brief, deferred to a separate milestone. They carry an accessible "(coming soon)" label and are not interactive.
- Delete row action is implemented inline (no separate delete page): activating it opens a confirm dialog (destructive-action confirm affordance) before the API call fires, per the design-reviewer's requirement that any destructive op carry a confirm step.
- Convention followed: `.project/design-system.md#Design system — data tables`

- [ ] Loading state: while product data loads, the table renders skeleton rows in place of the data rows.
- [ ] Empty state: when there are zero products, the page renders the shared EmptyState illustration with a primary "Add Product" CTA; because product creation lives behind the same deferred detail/edit milestone as View and Edit, the CTA renders in that same stubbed, non-interactive state (no navigation).

- [ ] Disabled/edge state: each row's View and Edit actions render visibly disabled and stubbed (no navigation), since the detail/edit pages are out of scope for this milestone. ## Design (recorded, consistent)
- Column set: the table's columns are the Products entity's standard fields (per the brief), with a final Actions column (View, Edit, Delete) appended.
- Sort/filter: every column except Actions carries independent sort and a column filter; Actions carries neither.
- Pagination: server-side, 30 rows per page, with a page-size control defaulting to 30.
- Loading: skeleton rows render in place of data rows while product data loads.
- Empty: shared EmptyState illustration plus a primary "Add Product" CTA. The CTA is stubbed and disabled (no navigation) because product creation lives behind the same deferred detail/edit milestone as View and Edit below, keeping the stub scope consistent across every entry point into the not-yet-built pages.
- Error: on fetch failure, reuse the shared DataTable component's own error-state rendering rather than a page-specific error UI, per the "reuse the shared DataTable component, do not hand-roll" rule.
- Row actions:
  - View and Edit render disabled and stubbed (aria-disabled="true", no navigation, no API call): the detail/edit pages are explicitly out of scope for this milestone per the brief.
  - Delete is fully implemented inline: clicking opens a confirm dialog (the destructive-action confirm affordance); confirming calls the delete API and removes the row from the table on success; on failure, an inline error notification surfaces and the row stays in place.
- Accessibility: sortable column headers are keyboard-operable and expose aria-sort; each column filter control carries an aria-label naming the column it filters; the Delete button carries an aria-label identifying the row's product (for example, "Delete <product name>"); the confirm dialog uses a dialog role, moves focus into itself on open, and returns focus to the triggering button on close; the stubbed View/Edit buttons carry aria-disabled="true" plus a label indicating they aren't yet available (for example, "View, coming soon") so they are not announced as interactive.

## Acceptance criteria

- [ ] Empty state: when the search returns zero rows, the table shows the existing shared `EmptyState` illustration with a "Clear search" call-to-action (in place of the entity's normal create CTA, since the table isn't actually empty of data) that clears the query and restores the full result set at page 1.
- [ ] Error / failure path: if the search request fails, the table body is replaced with an inline error message plus a "Retry" action that reissues the same query; the typed search term stays in the input so the user doesn't lose it.
- [ ] Disabled / edge state: the search input is disabled while the table is in its loading (skeleton-row) state, so a new query can't fire while a request is already in flight; a query that is empty or whitespace-only is treated as "no search" and clears the filter without an extra round trip.

## Design (recorded, consistent)
- The search input is a single, app-level control added to the shared `DataTable` component (`src/components/DataTable.tsx`), rendered once above the column headers, additive to (not a replacement for) the existing per-column sort/filter controls (`.project/design-system.md#Design system — data tables`).

- Search is server-side, issued on the same request cycle as the existing server-side pagination (`.project/design-system.md#Design system — data tables`: "server-side pagination at 30 rows per page"). It is not a client-side filter over the currently loaded page, since that would silently hide matches outside the loaded page.
- Loading state while a search request is in flight reuses the existing skeleton-row loading state; the search input is disabled for that duration (see acceptance criteria) to prevent overlapping requests.
- Empty-result state reuses the shared `EmptyState` illustration already mandated for tables with no rows, with its CTA swapped to "Clear search" for the no-match case (the entity create CTA doesn't apply here, since the table has data and the query simply didn't match any of it).

- Convention followed: `.project/design-system.md#Design system — data tables` (per-column sort/filter scope, server-side pagination, empty-state and loading-state patterns) and the existing shared component `src/components/DataTable.tsx`.
