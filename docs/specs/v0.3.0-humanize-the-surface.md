# milestone-feeder v0.3.0 — humanize the surface

**Status:** design spec (brainstormed 2026-06-18, locked). Feeds the dogfood
milestone: this document is the brief `plan` will consume to author v0.3.0 as a
GitHub milestone the driver then builds.

**Type:** breaking change vs v0.2.0 (command surface **and** config keys). No
back-compat shims, no deprecated aliases. Bumps to **v0.3.0**.

---

## 1. Theme

v0.2.0 works; its **surface** speaks in jargon. The verbs (`decompose`,
`refine`), the flag (`--apply`), and the config keys (`substrateDir`,
`selfCheck`) all name the *implementation*, not the user's intent. v0.3.0 is a
single, focused pass that makes everything a user sees, types, or reads speak
their language — built to the `writing-for-humans` standard, with the vocabulary
table (§12) as the backbone.

It is **not** a behavioral redesign. One behavioral change is in scope — and only
because it makes the surface honest: `create` now builds **exactly the plan you
approved** (§4), instead of re-running the pipeline and possibly diverging from
the preview. Everything else is rename + rewrite.

---

## 2. The command surface — verbs, not flags

Three intent-named verbs replace two verbs + a flag. Each verb *is* the
explanation of what it does.

| v0.2.0 | v0.3.0 | What the user means |
|---|---|---|
| `decompose <brief>` (preview) | **`plan <brief>`** | "Turn my idea into a reviewable plan." |
| `decompose <brief> --apply` | **`create <brief>`** | "Approve it — build the milestone + issues." |
| `refine <milestone>` | **`update <brief>`** | "My plan changed — sync it to the milestone." |
| `setup` | **`setup`** | "Set me up." (role unchanged; new key names) |

**Zero flags anywhere.** `--apply` is deleted from skills, agents, docs, README,
and the spec. The token-match / natural-language flag-recognition prose
(`skills/decompose/SKILL.md:22-29`, `skills/refine/SKILL.md:22-29`) is removed
entirely — there is nothing left to recognize. Claude Code does not argument-parse,
and a verb needs no parsing.

**No deprecated aliases.** The `decompose` and `refine` skill directories are
*renamed*, not shimmed. A v0.2.0 user who types `/milestone-feeder:decompose`
gets "no such command" — the honest signal for a breaking bump. The word
`decompose` appears **nowhere** in the repo after this release (§11 verifies).

### Skill directory moves

| v0.2.0 path | v0.3.0 path | Change |
|---|---|---|
| `skills/decompose/SKILL.md` | `skills/plan/SKILL.md` | rename + drop the apply path (moves to `create`) |
| — | `skills/create/SKILL.md` | **new** — reads the plan file, performs the GitHub writes |
| `skills/refine/SKILL.md` | `skills/update/SKILL.md` | rename + re-shape to plan-driven reconcile (§5) |
| `skills/setup/SKILL.md` | `skills/setup/SKILL.md` | in place; new key names |

## 3. The plan file is now a load-bearing contract

In v0.2.0 the plan file was a throwaway preview written as an audit record. In
v0.3.0 it is **the build artifact** — the single source of truth `create` and
`update` deploy. The mental model: **the plan file is the spec; GitHub is the
deployment.** `plan` compiles it; `create` deploys it fresh; `update` re-deploys
it onto an existing milestone.

`plan` writes it to the same gitignored scratch path, by a **deterministic slug**
of the milestone goal so `create`/`update` can find it from the same brief:

```
.milestone-feeder/plan-<slug>.md
```

### Plan-file contract (what `create`/`update` parse)

The format is essentially today's `§7-preview` plan file, formalized as a
contract. It MUST carry, unambiguously:

| Field | Why a consumer of the file needs it |
|---|---|
| **Milestone title** (exact) + one-line goal | `create`/`update` resolve the milestone by exact title (create-or-adopt). |
| **Milestone description** (wave order, local slugs `#A`/`#B`) | The wave-encoded description to PATCH after issue numbers exist. |
| **Per surviving issue:** slug, title, full §4 body, labels, surface/risk | The issues to create/patch — verbatim, no regeneration. |
| **Parked issues:** slug, title, kind (`product-gap` \| `needs-human-direction`) | Marked, **never created.** Routed to the needs-input report. |
| **Dropped issues:** slug, title, the parked dep that dropped them | Marked, **never created** (a dependent of a parked issue can't build). |
| **Self-check verdict line** | `create`/`update` report it; `create` *trusts* it (no re-vet). |
| **Source brief reference** | inline \| `file:<path>` \| `epic #<n>` — drives report routing + the brief↔plan match. |

The slug→`#n` rewrite still happens at write time (issue numbers don't exist
until creation) — that is the load-bearing two-pass mechanic from v0.2.0
`§7-apply.d`, preserved verbatim. What changes is **only** the source of the
issue bodies/labels/waves: read from the plan file, not regenerated.

## 4. `create` — deploy the approved plan (read-the-plan, faithful)

`create <brief>` honors the promise the quickstart makes: *"Once you approve, run
`create`."* What you read is what gets built.

```
Resolve the plan file for <brief> (deterministic slug)
  ├─ found    → read it; deploy EXACTLY it (no pipeline re-run, no re-vet —
  │             trust the recorded gate verdict)
  └─ absent   → run `plan` first (full pipeline + gate, writes the plan file),
                then deploy it
Deploy = §7-apply write sequence, unchanged:
  a. ensure labels (gh label create --force)
  b. create-or-adopt the milestone by exact title (env.t quote-safe resolve)
  c. create each SURVIVING issue (parked/dropped are NEVER created); build slug→#n map
  d. second pass — rewrite slug→#n in created-issue bodies + the milestone description
  e. file the needs-input report (epic comment if the brief was an epic; else local file)
```

- **Faithful:** on the found path, `create` does **not** re-dispatch the
  architect / issue-author, and does **not** re-run the self-check gate. The plan
  file already recorded the gate-surviving set; `create` writes it.
- **Standalone:** the absent path keeps `create <brief>` usable with no prior
  `plan` — it just runs `plan` for you first.
- **Staleness:** if the brief text changed since `plan` ran, the slug differs →
  no match → `create` re-plans (faithful: a changed brief earns a fresh plan). If
  the slug matches an older plan file, `create` uses it (you approved *that*) and
  prints a one-line "using existing plan file from <when> — re-run `plan` to
  refresh" notice.

The v0.2.0 `§7-apply` machinery (label ensure, create-or-adopt, two-pass
slug→`#n`, partial-failure/resume, report routing) moves wholesale into the
`create` skill. The gate (`§6`) moves into `plan`, reachable by `create` only via
the run-`plan`-first fallback.

## 5. `update` — reconcile a refreshed plan onto the existing milestone

`update <brief>` is for *"my plan changed."* You edit your brief, re-run `plan`
to refresh the plan file, then `update` reconciles it against the live milestone.
It absorbs refine's "re-vet in place" job **for free** — the plan going in is
always gate-clean, so reconciling it inherently repairs any issue that drifted
from spec.

```
Refresh/resolve the plan file for <brief>   (same slug logic as create; run `plan` if absent)
Resolve the existing milestone by exact title
  └─ NOT FOUND → 🔴 error-and-stop: "update re-deploys onto an existing milestone —
                  none titled "<t>". Run /milestone-feeder:create first."
Reconcile plan file vs live milestone (match by exact issue title):
  in plan, not on GitHub        → CREATE it
  in both, body differs         → PATCH it to match the plan (show the diff)
  new dependency edge           → ADD it; re-render the wave description; PATCH it
  on GitHub, not in plan        → 🔴 FLAG for your decision  (NEVER close/delete)
  nothing differs               → NO-OP (zero gh writes; say so)
```

- **Never closes, never deletes.** Refine's invariant holds. A live issue absent
  from your refreshed plan is flagged in the report, not auto-closed — closing is
  your call (consistent with "park, don't guess").
- **Idempotent.** Re-running on an already-synced milestone writes nothing.
- **Plan is the source of truth.** `update` patches a live body to match the plan
  and **shows the diff** before applying (the announce-then-write checkpoint) — it
  never silently clobbers. If you want a different issue, change your brief/plan,
  not GitHub.
- **Pure preview is `plan`.** There is no `update`-specific preview mode (no
  flags). `plan` is the preview for the whole family; `create`/`update` announce
  what they will write, then write.

`update` reuses `create`'s write primitives (label ensure, `gh issue create`,
`gh issue edit --body`/`--add-label`, the `gh api PATCH` description form,
slug→`#n` for any newly-created issues) by reference — no second definition, the
same discipline refine already uses against decompose.

## 6. Config — rename the keys, keep the file

The file stays **`.milestone-config/feeder.json`** (pairs with `driver.json`,
matches the plugin name, avoids hook-script churn). The **keys inside** — the part
a user reads and edits — are humanized.

| v0.2.0 key | v0.3.0 key | Type / value | Note |
|---|---|---|---|
| `substrateDir` | **`projectDocs`** | string (path), default `.project/` | Where your project's standing docs live. |
| `selfCheck` | **`reviewer`** | `"milestone-driver"` \| `"internal"` \| `false` | Who checks each issue before it's created; `false` = off. |
| `issueSizeGuidance` | **`issueSize`** | string, e.g. `"≤1 PR, ≤1 new screen"` | Optional sizing rule. |
| `decomposerAgent` | **`architectAgent`** | string, default `milestone-feeder:architect` | Tracks the agent rename (§7). |
| `issueAuthorAgent` | `issueAuthorAgent` | string, default `milestone-feeder:issue-author` | **Unchanged** — issue-author isn't purged. |
| `sourceGlobs` (self-protection) | `sourceGlobs` | string[] | **Unchanged** — developer-only; matches the driver; read by the hook. |

**Migration is a non-event for this repo:** the dogfood `feeder.json` holds only
`sourceGlobs`, which is unchanged — nothing here breaks. An external consumer who
set `substrateDir`/`selfCheck` re-runs `setup`; the skills read only the new keys
(no silent old-key fallback). `setup` MAY detect an old-key file and offer to
migrate it in place, with a notice — a courtesy, not a contract.

**`reviewer` semantics are identical to `selfCheck`'s** — only the name changes.
The `false` value reads slightly oddly ("the reviewer is false" = no reviewer);
that is the accepted trade for a short noun key.

## 7. Agent — `decomposer` → `architect`

The breakdown agent (brief → candidate issues + dependency graph + wave order) is
renamed. The SPEC already calls it the "architect lens," and it pairs cleanly:
**the architect designs the breakdown, the issue-author writes each issue.**

| Artifact | v0.2.0 | v0.3.0 |
|---|---|---|
| Agent file | `agents/decomposer.md` | `agents/architect.md` |
| Agent id | `milestone-feeder:decomposer` | `milestone-feeder:architect` |
| Config key | `decomposerAgent` | `architectAgent` |
| Default value | `milestone-feeder:decomposer` | `milestone-feeder:architect` |

The agent's *contents* (its "Architect lens" framing, structured return block,
clauses) need only the self-reference and any "decompose" wording updated. The
`issue-author` agent is untouched.

## 8. README rewrite

Rewritten to the `writing-for-humans` standard. Section order (locked):
**Quick start → Before you start → How it works.** Lead with the quick win, then
the prerequisites, then the conceptual model.

### Before you start — the real permissions

Enumerate exactly what a **consumer** must grant, derived from the `gh` surface
the skills actually invoke (not this repo's dogfood `settings.local.json`, which
only carries `Read` globs):

- `gh` installed and authenticated, operating in a directory connected to a
  GitHub repo — *"otherwise we won't be able to create your milestone or issues."*
- Claude permitted to run the write commands `create`/`update`/`setup` use:
  `gh label create`, `gh api … /milestones` (POST/PATCH),
  `gh issue create`/`edit`/`list`/`view`/`comment`, and `Read` for your project
  docs. (`plan` is read-only on GitHub — it only `gh issue view`s an epic brief
  if you hand it one; everything else it writes is a local file.)
- git; bash + `jq` (or PowerShell 7+) for the `no-source-edit` hook.
- `milestone-driver` is **optional** — it backs the default `reviewer:
  "milestone-driver"`; absent, the gate degrades to `"internal"`.

### How it works — make the guardrail legible (#4)

Describe the grounding + gate in plain English, *no machinery added*:

> You hand `plan` your idea. It reads your project's standing docs and breaks the
> idea into small, buildable issues — recording, for each one, the decisions an
> engineer would otherwise have to invent, and **citing the house rule it
> followed**. Before it finishes, it **checks each issue against your conventions**
> and fixes or flags anything that doesn't hold up. A decision with no
> conventional default it never guesses — it flags it for you. Then `create`
> builds exactly what you approved.

This is the existing "substrate-grounding + self-check gate" mechanism, named in
words a newcomer owns. Nothing new is built — #4 resolves to legibility, not
machinery.

## 10. Harness — migrate + re-run (in scope)

The credibility harness is the proof the rename didn't change behavior, so it
moves with the surface:

- **Scenario inputs** (`tests/scenarios/*/brief.md`, `feeder-env.md`,
  `expected.md`) → new verbs (`plan`/`create`/`update`), new keys
  (`projectDocs`/`reviewer`), `feeder.json` kept.
- **Harness docs** (`tests/README.md`, `tests/RESULTS.md`) → new vocab.
- **Re-run** every scenario on the v0.3.0 surface → fresh transcripts
  (`run-log.md`, `run-output.md`) so the green scorecard reflects v0.3.0, not
  v0.2.0. Historical v0.2.0 transcripts are replaced by the re-run, not
  hand-edited.

Result: **4/4 green stays true and provable** after the rename.
