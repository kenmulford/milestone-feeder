# v0.7.0 — Capability-aware completeness (implied surfaces)

**Status:** design brief, pending dogfood (`/milestone-feeder:plan`).
**Seed issue:** #176.
**Date:** 2026-06-26.

> This is a *brief* — the human-authored input. The buildable issue set (the §4
> specs) is produced by dogfooding the feeder (architect → issue-author →
> self-check gate), not hand-authored here.

---

## The problem

A brief names a capability — "add email", "user management", "sync" — and that
name quietly commits to a standard set of surfaces (screens, actions, endpoints,
jobs, settings) the brief never spells out. Today the architect breaks down only
what's literally written, so the implied surfaces get discovered one-at-a-time
mid-build and ship as unplanned rework. The canonical smell: an issue set that
says "here's a Users page" with no CRUD, no activate/deactivate, no
reset-password — the obvious companions missing not on purpose, but by omission.

The problem is real and named in the literature (implicit / tacit requirements);
the validated prior art for "a named capability implies a predictable companion
set" includes Withall's requirement-pattern "Extra Requirements", DDD's
repository-per-aggregate operation set, and the "split by CRUD operations"
story-splitting pattern. We are not inventing a new idea — we are giving the
architect a disciplined version of one teams already do tacitly.

## The scope boundary (keystone)

**This is the feeder, not the driver.** The feeder's job ends at *naming the
right surfaces as issues*. The **driver** fleshes each issue out into code
against the conventions, standards, and frameworks recorded in `.project`. So:

- Feeder (this feature): ensure the issue set includes the standard *surfaces* a
  capability implies — at **issue / surface granularity**, conceptual, short.
- Driver (already does this): build each issue to `.project` convention — CRUD
  wiring, validation, the framework's standard patterns — at **code granularity**.

We can't (and won't) automate every feature detail. We can do better than "here's
a Users page" with the activate/deactivate/reset-password concepts silently
absent. The implied-surface lists are therefore deliberately **conceptual and
short**, not exhaustive implementation checklists — which is also what avoids the
"false completeness" failure mode (see Non-goals).

## What it does

During breakdown, the architect consults a curated **implied-surfaces reference**
(a *reasoning prompt*, keyed by capability and by "new entity"). For each
capability or new entity the brief invokes, it considers the standard surfaces
that capability implies and ensures each is accounted for in the plan — sorting
each surface with the judgment it **already has** (grounded-vs-product-gap):

1. **Conventional surface** (standard plumbing, has a conventional default —
   e.g. email → a delivery-failure log; Users → reset-password) → proposed as a
   **default-in candidate in the reviewed plan**, labeled "implied — review /
   trim / augment".
2. **Genuine product-call surface** (no conventional default — e.g. email →
   suppression/unsubscribe policy, which depends on transactional-vs-marketing,
   jurisdiction) → **flagged as a decision** via the existing `PRODUCT_GAPS`
   path, never silently pre-included.

The architect already knows how to tell these apart — it is the same
grounded-vs-gap call it makes today. The reference only ensures it *considers*
the full conventional set rather than failing to ask.

### Why default-in is correct here

Default-in surfaces are proposed into a **plan the human is required to review
and approve before any issue is created** — not into shipped code. The cost of a
wrongly-included surface is "delete a line before approving," fully reversible.
(The over-acceptance evidence against opt-out defaults comes from consequential,
irreversible, *unreviewed* settings — organ-donation, cookie consent — and does
not transfer to a mandatory-review artifact.) This is a deliberate, conscious
evolution of the architect's contract, not a silent one.

### Anti-fixation (structural, not hoped-for)

Showing a partial list can suppress a reviewer's own recall of what's missing
(part-set cuing). So the review **explicitly** prompts: *"this is a starting set
for YOUR app — what's missing?"* — making "it missed these 2-3, add them" a
built-in step, not something we hope the human spontaneously does. The list is a
robust **start**, never a ceiling.

## The reference

- A curated, **PR-able** reference shipped with the plugin (so the capability
  list grows by community PR — the original goal), expressing each capability →
  its implied surfaces in stack-agnostic terms (screen / endpoint / job /
  setting; the architect maps to the project's stack).
- An optional **project-local overlay** in `.milestone-config/`, merged
  **additively** (a project can add capabilities and extend existing ones; it
  can't delete a global surface — per-run trimming is the plan review's job).
  Captures domain-specific clusters a universal reference can't hold (e.g. a
  church app's "giving": recurring schedules, statements, refunds, gateway
  config).
- It is a **reasoning prompt / floor**, NOT a rigid scope-catalog that emits
  candidates without review. The distinction matters: it stops the architect
  *failing to ask*; it does not *commit* scope on its own.

### Triggers

1. **New entity** — entities the brief introduces; fans out the baseline cluster
   (list / detail / create / edit / delete / states / permissions / audit) **per
   entity**, so "create 10 entity types" never ships ten bare pages.
2. **Named capability** — concept-matched (not just keyword) against the
   reference: "let admins message members" → the email/messaging cluster.
3. **Cross-cutting** — app-level (search/filter/sort, background jobs), not
   per-entity.

### States

States (empty / loading / error / unauthorized) are **considered** as surfaces
so they are never overlooked, but they land as **acceptance criteria inside their
screen issue** — where the issue-author already enumerates them — not as
standalone issues. ("States are features, not grain": surface them; let their
grain fall out naturally.)

## Grain & volume — reuse, don't rebuild

Granularity is **not** this feature's job. Fanned-out surfaces flow into the
architect's existing "~1 PR each" sizing. If a fleshed-out capability genuinely
crosses release boundaries, the existing v0.6.0 roadmap split handles it. (Note:
that split triggers on release boundaries / distinct deliverables, not on raw
fan-out volume — so the implied-surface lists must stay conceptual and short by
design, per the scope boundary above, rather than relying on the roadmap split to
absorb a flood.)

## Discovery for existing users

The project-local overlay is a new optional config file, so `feeder:setup` and
`update` mention it and the docs explain it — existing users get a path to
discover it (the standing existing-user-discovery rule).

## Non-goals / explicitly NOT this feature

- **Exhaustive fan-out.** A curated, conceptual, short set — reviewed and
  augmentable — never an auto-dump of every conceivable surface.
- **Silent scope expansion.** Every proposed surface is reviewed; genuine
  product-calls are parked as decisions, not pre-included.
- **Code-detail specification.** That's the driver's job, against `.project`.
- **A rigid global scope-catalog** that emits candidates without review, or that
  rots into an unbounded maintained list.
- **States as standalone issues.**
- **Extending the brief-coverage-verifier** to audit implied surfaces — it is
  brief-anchored by contract; out of scope for 0.7.0.

## Contract change (do consciously)

This adds a labeled "implied surface" disposition to the architect's breakdown
that is distinct from both a grounded candidate and a parked product-gap. That is
a deliberate change to the architect's contract and its rigor gate — to be made
consciously, with the never-invent-product-scope boundary preserved: conventional
surfaces are *proposed for review*, genuine product-calls still *park*.
